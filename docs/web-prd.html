1. Goals & non-goals

Goals

  • Ship a standalone marketing site for Blihop / blihops.com
  • Full public page set (services, about, pilot, content, careers, product teasers, legal, contact)
  • Static-first content so the site works with no backend
  • Clear boundary so a future API can replace mocks without rewriting pages
  • Polished, modern UI with intentional motion — redesign freely
  • Strong SEO defaults (metadata, sitemap, robots, semantic HTML)

Non-goals (this repo)

  • Admin CMS UI
  • Backend / database / email / media storage
  • Auth / JWT / admin sessions
  • Full Skills or Talent products (teaser pages + outbound links only)
  • Pixel-perfect port of the monorepo site
  • Monorepo package dependencies (@repo/*)

3. Architecture

Approach A — App Router + content modules + API boundary. Pages stay thin. Data and form submissions go through lib/api and lib/forms. Today those modules use static/mock implementations; later they call the real backend with the same function shapes.

Pages / sections

Render UI only. Call data helpers and form submitters.

→
lib/api + content/

Mock GETs from static content. Later: HTTP to API.

→
lib/forms

Validate → mock POST (log + success). Later: real POST.

Rules

  • Pages must not import raw content files if an lib/api helper exists — always go through the boundary.
  • No secrets in the web app. Env vars are public site config only (e.g. site URL, product app URLs).
  • Components are presentational; business/data logic lives in lib/.
  • Prefer server components for static pages; client components only for forms, motion, and interactive UI.

5. Content model

v1 content lives in the repo under content/. Shape it like a future CMS so swapping to API responses is a mapping change, not a redesign.

Marketing copy

  • Site-wide: nav labels, footer, brand, CTAs
  • Per-page: heroes, sections, FAQs
  • Legal: privacy, terms

CMS-shaped (mock)

  • Insight — title, slug, excerpt, body (HTML or MD), cover, publishedAt, SEO
  • Case study — same core fields + outcome metrics if needed
  • Open role — title, slug, location, type, description, active
  • Testimonial / partner — optional homepage social proof

API boundary (example signatures)

// lib/api — same shapes now and later getInsights(): Promise<InsightSummary[]> getInsightBySlug(slug: string): Promise<Insight | null> getCaseStudies(): Promise<CaseStudySummary[]> getCaseStudyBySlug(slug: string): Promise<CaseStudy | null> getOpenRoles(): Promise<OpenRole[]> getTestimonials(): Promise<Testimonial[]> getPartners(): Promise<Partner[]>

Implementations: v1 read from content/. Future: fetch against the backend with the same return types.

GET vs POST (v1): all reads use mock/static data. All writes use a mock request body, console.log (or equivalent dev logger), and show a clear success state. No real persistence until the API exists.

8. Success criteria

  • Repo is standalone — no monorepo or @repo/* dependencies
  • Every listed v1 route works and is linked from nav where appropriate
  • Content and forms go through lib/api / lib/forms only
  • Mock GETs and mock POSTs behave as specified (log + success for writes)
  • Mobile-first responsive layout; polished motion without 3D bloat
  • SEO basics: titles, descriptions, sitemap, robots, sensible headings
  • Lighthouse-friendly defaults (no unused heavy deps)
